# Agent & Chat Architecture Review

> **Principal Engineer Review** | March 2026
>
> Assessment of the AgentBank chat and agent interaction architecture against production best practices for the Anthropic API, multi-turn conversation design, and agentic systems. Based on current documentation, community knowledge, and known API behaviours.

---

## Executive Summary

The architecture is well-designed for a POC with strong fundamentals: hexagonal ports/adapters, two-phase confirmation for write safety, tool gating by onboarding state, and a clear streaming pipeline. However, several gaps could cause subtle-but-serious issues in implementation. The most critical: **financial figures flowing through Claude's text generation** (where it can hallucinate amounts), **missing `tool_result` handling for the Anthropic API contract**, **no prompt caching strategy** (leaving significant cost and latency gains on the table), and **no circuit breaker for the Anthropic API itself** (529 overloaded errors are frequent and will silently kill streams).

## MUST-FIX Issues

### MF-1: Financial figures must never be generated by the LLM

**Severity:** Critical
**Location:** system-architecture.md §3.4 (streaming diagram), api-design.md §3.4 (UIComponent types)

**Problem:** The streaming diagram shows Claude generating text like `"Your bal..."` *after* receiving a tool result containing `balance: 1247.50`. Claude is narrating the balance in natural language. This means the user sees `£1,247.50` as LLM-generated text — and LLMs hallucinate numbers. Claude could say "Your balance is £1,247" (rounding), "£1,247.05" (transposing digits), or invent a completely different number if the context window is noisy.

The architecture already has the right escape hatch — `ui_components` with typed cards (BalanceCard, TransactionListCard). But there's no documented rule that **Claude must never re-state financial figures in text**. The system prompt's `CARD_RENDERING_RULES` and `SAFETY_RULES` blocks need an explicit directive.

**Fix:**
1. Add to `SAFETY_RULES` in the system prompt: *"NEVER repeat financial amounts, balances, account numbers, or transaction details in your text response. Always use the appropriate card component (BalanceCard, TransactionListCard, etc.) to display financial data. Your text should reference the card: 'Here's your current balance' — not 'Your balance is £1,247.50'. The card data comes directly from the bank; your text is generated and may be inaccurate."*
2. Add a server-side validation layer: if Claude's text response contains a currency pattern (`£X,XXX.XX`) that doesn't match any tool result in the same turn, log a warning and optionally strip it.
3. Document this as an architectural invariant alongside the two-phase confirmation invariant.

**Evidence:** MIT thesis on LLM hallucination in banking (2025) documents that financial data has a "shelf life measured in hours or minutes" while LLM training data has a "shelf life measured in years." Anthropic's own financial services documentation recommends structured outputs for all numerical data.

### MF-2: `respond_to_user` tool breaks the Anthropic API contract

**Severity:** Critical
**Location:** system-architecture.md §3.4, lines 416-423

**Problem:** The spec says: *"No tool_result is sent back to Claude (the tool call terminates the agent loop)."* This violates the Anthropic API contract. Every `tool_use` content block **must** have a corresponding `tool_result` in the conversation history. If you persist the conversation (which you do, in `content_blocks JSONB`) and load it for the next turn, Claude will receive a `tool_use` without a matching `tool_result`, causing a **400 error** on the next API call.

This is the #1 most common Anthropic tool-use bug in production (GitHub issues #25959, #9103). The Anthropic docs state: *"Each tool_use block must have a corresponding tool_result block."*

**Fix:** Two options:
- **(a) Synthetic tool_result:** After intercepting `respond_to_user`, persist a synthetic `tool_result` content block: `{ type: "tool_result", tool_use_id: "...", content: "Response delivered to user." }`. This satisfies the API contract on the next turn.
- **(b) Don't persist respond_to_user as a tool_use:** Strip the `respond_to_user` tool_use from `content_blocks` before persisting. Instead, persist the text and ui_components directly as assistant content. This is cleaner but requires more transformation logic.

**Recommendation:** Option (a) — simpler, safer, handles edge cases where the stream dies mid-respond_to_user.

### MF-3: No Anthropic API error handling for 529 (overloaded)

**Severity:** Critical
**Location:** system-architecture.md §9 (Error Handling Strategy)

**Problem:** The error handling table covers HTTP 400/401/403/404/429/500/502 but has no mention of **HTTP 529** (Anthropic overloaded). This is the most commonly encountered production error with the Anthropic API. It occurs when Anthropic's servers are under load and is **not your fault** (unlike 429 rate limits). Community reports indicate it's especially frequent during peak hours and with Opus/extended thinking.

Critically, 529 can occur **mid-stream**. After the initial 200 response, the SSE stream can emit an `error` event or simply hang (GitHub issue #867: clients hang indefinitely when streams stop sending data after thinking blocks complete — no close, no error, no timeout).

**Fix:**
1. Add 529 to the error handling table: `| Anthropic Overloaded | 529 | Retry with backoff | Claude API at capacity |`
2. Implement retry logic for the Claude API call itself (not just the SSE stream): 3 retries with exponential backoff (2s, 4s, 8s) and jitter.
3. Add a server-side timeout for the Claude API stream: if no event received for 30 seconds (separate from the client-side 15s heartbeat timeout), cancel the stream and retry.
4. On final retry failure, return an error card to the client: *"I'm temporarily unavailable. Please try again in a moment."*

**Evidence:** Multiple GitHub issues document this: anthropic-sdk-typescript #867 (hanging streams), #842 (interrupted mid-transmission). The Anthropic error docs distinguish 429 (your fault) from 529 (their fault) and recommend different retry strategies.

### MF-4: Summarisation runs inline and blocks the stream

**Severity:** High
**Location:** api-design.md §2.1 flow step 4, tech-decisions.md ADR-05

**Problem:** Step 4 of the chat flow says: *"Load message history (with summarisation if > 80 messages)."* This happens **before** streaming starts. The summarisation calls `claude-haiku-4-5` — a separate API call that takes 500-2000ms. Combined with steps 2-3 (JWT validation, conversation loading), the user could wait 2-3 seconds before seeing even the `thinking` event.

The spec acknowledges this: *"Conversation summarisation (step 4) runs inline for now but should be moved to a post-response background job if latency exceeds 2 seconds."* But "for now" in a POC has a way of becoming permanent.

**Fix:** Move summarisation to **after** the response completes, not before. The flow should be:
1. Emit `thinking` event
2. Load conversation history (raw, unsummarised)
3. If message count > 80, **include all messages** for this turn (Claude handles 200K context)
4. Stream the response
5. **After** `done`, run summarisation as a background job and update the conversation record

This changes the summarisation from "pre-flight compression" to "post-flight cleanup." The next turn loads the summarised history. Trade-off: the first turn after 80 messages uses more tokens (and costs slightly more), but the user gets sub-100ms TTFT consistently.

**Alternative:** If cost is a concern, summarise eagerly but **cache the summary**. Only re-summarise when new messages have been added since the last summary. Don't block the stream.

### MF-5: No prompt caching — missing 90% cost reduction

**Severity:** High
**Location:** system-architecture.md §3.2 (System Prompt Structure), tech-decisions.md ADR-10

**Problem:** The system prompt is reconstructed per request from 8 blocks (persona, user context, time context, rules, etc.). Of these, **5-6 blocks are completely static** across all users and all requests (PERSONA_BLOCK, TOOL_USAGE_RULES, CONVERSATION_RULES, CARD_RENDERING_RULES, SAFETY_RULES, ONBOARDING_RULES). Only USER_CONTEXT_BLOCK and TIME_CONTEXT_BLOCK change per request.

Without prompt caching, every request pays full input price for the static portion. For a system prompt estimated at 2,000-4,000 tokens, this is a 90% cost reduction left on the table. Prompt caching also reduces TTFT by 40-60% for subsequent requests.

**Fix:**
1. Structure the system prompt with static blocks first, dynamic blocks last.
2. Add `cache_control: { type: "ephemeral" }` to the last static block.
3. Move tool definitions into the system prompt (they're static per session) and cache them.

```typescript
const messages = [
  {
    role: "user",
    content: [
      { type: "text", text: STATIC_SYSTEM_PROMPT },  // Persona, rules, tools
      { type: "text", text: STATIC_SYSTEM_PROMPT_2, cache_control: { type: "ephemeral" } },
      { type: "text", text: DYNAMIC_CONTEXT },  // User name, time, onboarding step
    ]
  }
];
```

Wait — Anthropic's system prompt is a top-level field, not a message. Caching works on the `system` field:

```typescript
const response = await anthropic.messages.create({
  system: [
    { type: "text", text: STATIC_RULES },
    { type: "text", text: STATIC_RULES_2, cache_control: { type: "ephemeral" } },
    { type: "text", text: DYNAMIC_USER_CONTEXT },
  ],
  messages: [...],
});
```

**Evidence:** Real-world case study: $720/month to $72/month (90% reduction) with prompt caching. Anthropic docs confirm cache_control works on system prompt blocks. Minimum cacheable length for Sonnet is 1,024 tokens.

### MF-6: No `max_tokens` specified — required by Anthropic API

**Severity:** High
**Location:** system-architecture.md §3.1 (Agent Loop), api-design.md §2.1

**Problem:** The architecture doesn't specify `max_tokens` for the Claude API call. Unlike OpenAI where it's optional, **Anthropic requires `max_tokens` as a mandatory parameter**. If omitted, the SDK will throw a validation error before the request is sent.

More importantly, without a thoughtful `max_tokens` value, the agent loop has no output budget. A single response could theoretically consume the full 64K output token limit, blocking the stream for minutes and incurring significant cost.

**Fix:**
1. Set `max_tokens: 4096` for normal chat responses (sufficient for text + tool calls).
2. Set `max_tokens: 1024` for the Haiku summarisation call (ADR-05 already says `max_tokens: 500` — good).
3. Document these values in the system-architecture.md agent loop section.
4. Consider `max_tokens: 8192` if Claude needs to emit multiple tool calls in a single response (parallel tool calling with complex parameters).

## HIGH Priority Issues

### H-1: Missing `is_error` flag on tool error results

**Severity:** High
**Location:** system-architecture.md §9.2 (Error Flow), §3.3 (Tool Registry)

**Problem:** The `ToolResult` interface has `success: boolean` and `error?: ToolError`, but the architecture doesn't show how this maps to the Anthropic API's `tool_result` format. Anthropic's API has a specific `is_error: true` field on tool_result content blocks. If you return an error as a normal tool_result (without `is_error`), Claude treats it as a successful result that happens to contain error-looking text — and may try to interpret the error as data.

**Fix:** When converting ToolResult to Anthropic tool_result format:
```typescript
// In the agent loop, when feeding tool results back to Claude:
const toolResultBlock = {
  type: "tool_result",
  tool_use_id: toolUseBlock.id,
  content: result.success
    ? JSON.stringify(result.data)
    : result.error.message,
  is_error: !result.success,  // CRITICAL: tells Claude this was an error
};
```

### H-2: Parallel tool calls not handled

**Severity:** High
**Location:** system-architecture.md §3.1 (Agent Loop diagram)

**Problem:** The agent loop diagram shows a single tool_use → execute → loop cycle. But Claude (Sonnet 4.6) frequently emits **multiple tool_use blocks in a single response** — especially for queries like "What's my balance and show my recent transactions?" Claude will call both `check_balance` and `get_transactions` in parallel.

The architecture must handle this: all tool_use blocks in a single assistant message must have their corresponding tool_result blocks returned in a single user message. Returning them one at a time (separate user messages) violates the API contract.

**Fix:**
1. Document that the agent loop must collect **all** tool_use blocks from a single assistant response.
2. Execute them concurrently (they're independent reads in most cases).
3. Return all tool_result blocks in one user message array.
4. Stream `tool_start` / `tool_result` events to the client for each one.

```typescript
// Collect all tool calls from this assistant turn
const toolUseBlocks = response.content.filter(b => b.type === 'tool_use');

// Execute all concurrently
const results = await Promise.all(
  toolUseBlocks.map(block => registry.execute(block, context))
);

// Send all results back in one user message
const toolResultBlocks = toolUseBlocks.map((block, i) => ({
  type: 'tool_result',
  tool_use_id: block.id,
  content: JSON.stringify(results[i].data),
  is_error: !results[i].success,
}));
```

### H-3: `content_blocks` storage format is underspecified

**Severity:** High
**Location:** data-model.md §2.3 (messages table)

**Problem:** The `content_blocks JSONB` column stores *"Structured Anthropic MessageParam blocks"* — but the schema doesn't specify whether these are the raw Anthropic API format or a transformed format. This matters because:

1. Anthropic's content block format includes `type` discriminators (`text`, `tool_use`, `tool_result`, `thinking`) that must be preserved exactly.
2. If you store transformed/simplified blocks, you need a reconstruction step before feeding them back to Claude.
3. Tool call IDs (`toolu_XXXX`) are ephemeral per conversation turn — they must be preserved in storage to maintain the tool_use → tool_result linkage.

**Fix:** Document explicitly:
- `content_blocks` stores the **exact Anthropic API format** (array of content blocks with `type`, `id`, `name`, `input` for tool_use; `tool_use_id`, `content`, `is_error` for tool_result).
- Tool call IDs must be preserved.
- When loading history for the next turn, `content_blocks` are fed back to Claude as-is.
- `content` (TEXT column) is the human-readable summary (for search, display in conversation list) — not used for the API.

### H-4: Token counting and context budget not specified

**Severity:** High
**Location:** system-architecture.md §3.1, tech-decisions.md ADR-05

**Problem:** The architecture specifies summarisation at 80 messages but doesn't account for **token count**. A conversation with 50 messages could already exceed the practical context budget if messages contain large tool results (a transaction list with 50 items, a spending breakdown with 10 categories). The 80-message threshold is a proxy for tokens, but a poor one.

Additionally, tool definitions themselves consume significant context. With 40+ tools, each with descriptions and JSON schemas, the tool definitions alone could be 10,000-20,000 tokens.

**Fix:**
1. Use Anthropic's free `count_tokens` endpoint to track actual token usage per conversation.
2. Set a token budget (e.g., 80K of the 200K context window) and trigger summarisation based on that, not message count.
3. Consider the Anthropic beta `context-management-2025-06-27` header, which automatically clears old tool results as you approach token limits.
4. Track tool definition token cost at startup and subtract from the available budget.

### H-5: No idempotency on the agent loop retry

**Severity:** High
**Location:** system-architecture.md §3.5 (Stream Recovery)

**Problem:** The retry spec says: *"On retry, the client sends the same `conversation_id`. The server detects the conversation already has the user's latest message persisted, skips re-persisting it, and re-runs the agent loop from the last persisted assistant message."*

But if the original request partially executed tools (e.g., `check_balance` succeeded but the stream died before Claude finished), the retry will re-run the agent loop from scratch. Claude may call `check_balance` again — harmless for reads, but what about writes? If the pending_action was already created, a retry could create a duplicate pending_action. The `idempotency_key` on pending_actions helps, but the spec doesn't show how it's generated or enforced during retries.

**Fix:**
1. Document the idempotency key generation strategy: `${conversation_id}:${message_index}:${tool_name}:${params_hash}`.
2. Before creating a pending_action, check for an existing one with the same idempotency_key.
3. For read tools: re-execution is safe (idempotent by nature).
4. For the Claude API call itself: the same input messages will produce different outputs (non-deterministic). Accept this — the retry is "close enough," not exact replay.

### H-6: Streaming `input_json_delta` parsing risk

**Severity:** High
**Location:** system-architecture.md §3.4 (Streaming)

**Problem:** When Claude streams a tool_use block, the tool input parameters arrive as `input_json_delta` events — **partial JSON fragments that are not valid JSON until the block is complete**. The architecture shows the server intercepting tool_use at `content_block_start`, but doesn't specify how to handle the incremental JSON.

If the server tries to parse tool input before `content_block_stop`, it will get a JSON parse error. This is especially dangerous with fine-grained tool streaming (available for Sonnet 4.6), where deltas arrive without buffering.

**Fix:**
1. Document that tool input JSON must be **accumulated** across all `input_json_delta` events.
2. Only parse the complete JSON at `content_block_stop`.
3. Handle the case where `max_tokens` truncates the JSON mid-stream — you'll get invalid JSON with no `content_block_stop`. Detect this via `message_delta.stop_reason === 'max_tokens'` and return an error card.

## NICE-TO-HAVE Improvements

### N-1: Consider `tool_choice` for specific flows

**Location:** system-architecture.md §3.1

The architecture uses the default `tool_choice: "auto"` for all requests. For the `__app_open__` flow, consider `tool_choice: { type: "tool", name: "respond_to_user" }` to force Claude to generate the greeting card directly, skipping unnecessary tool calls (Claude sometimes calls `check_balance` before greeting). Trade-off: forced tool choice is incompatible with extended thinking.

### N-2: Add token usage tracking per conversation

**Location:** system-architecture.md §10.2 (Observability)

Log `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, and `cache_read_input_tokens` from every Claude API response. This enables cost tracking per user, per conversation, and early detection of runaway loops.

### N-3: Consider hard-coding deterministic responses

**Location:** system-architecture.md §4 (Two-Phase Confirmation)

Confirmation success/failure messages don't need an LLM. When a payment confirms successfully, the response could be a hard-coded SuccessCard + template text ("£50 sent to James") instead of another Claude API call. This saves a round-trip, reduces cost, and eliminates hallucination risk for the most critical moment in the flow.

### N-4: `stop_reason: "pause_turn"` handling

**Location:** system-architecture.md §3.1

Anthropic's server-side tools have a 10-iteration limit. When hit, the API returns `stop_reason: "pause_turn"`. The architecture's `MAX_TOOL_ITERATIONS` of 5 should prevent this, but if a future change raises the limit, the agent loop needs to handle `pause_turn` by sending the response back for continuation. Document this stop reason alongside the existing `end_turn` / `max_tokens` handling.

### N-5: System prompt should include a `do not invent` directive

**Location:** system-architecture.md §3.2

Add to SAFETY_RULES: *"If a tool call fails or returns no data, say so honestly. Never invent account numbers, sort codes, transaction references, dates, or amounts. If you don't have the data, tell the user and offer to try again."* This is a well-established anti-hallucination pattern for financial AI.

### N-6: Structured outputs beta for tool schemas

**Location:** tech-decisions.md, api-design.md §3

Anthropic's structured outputs beta (`anthropic-beta: structured-outputs-2025-11-13`) with `strict: true` on tool definitions guarantees schema validation on tool inputs during inference. This eliminates malformed tool parameters. Available for Sonnet 4.5+. Evaluate for Foundation — it's a low-effort, high-value safety net. Caveat: GitHub issue #885 reports 500 errors with complex nested schemas and `strict: true`, so test thoroughly.

### N-7: Rate limit of 10 req/min may be too low

**Location:** api-design.md §2.1

10 requests/minute per user means one message every 6 seconds. In an active banking conversation ("check balance" → "show transactions" → "send £50 to James" → "confirm"), a user could hit this limit in under a minute. Consider 20-30 req/min for chat, separate from the API read endpoints.

## Trade-Off Decisions for the Team

### T-1: Summarisation model — Haiku vs. server-side compaction

ADR-05 uses Haiku for summarisation. Anthropic now offers **server-side compaction** (beta for Opus 4.6, likely coming to Sonnet). This would eliminate the custom summarisation code entirely. Trade-off: it's beta, Opus-only for now, and you lose control over what gets summarised. **Recommendation:** Keep Haiku summarisation for POC, but track the server-side compaction beta for post-POC.

### T-2: 40+ tool definitions — discovery vs. always-loaded

With 40+ tools, tool definitions consume 10K-20K tokens per request. Two strategies:
- **(a) Always load all tools:** Simple, but expensive and may confuse Claude with irrelevant options (lending tools when the user hasn't completed onboarding).
- **(b) Dynamic tool loading:** Only load tools relevant to the current context (onboarding tools during onboarding, banking tools after). Reduces tokens and improves Claude's tool selection accuracy.

The architecture already has onboarding gating — extend this to contextual gating (e.g., don't load lending tools unless the user asks about loans). Trade-off: adds complexity, but could halve the tool definition token cost.

### T-3: `respond_to_user` as the only exit — fragile stopping condition

The agent loop terminates when Claude calls `respond_to_user`. But what if Claude generates a pure text response without calling any tool (including `respond_to_user`)? With `tool_choice: "auto"`, this is possible. The `stop_reason` would be `end_turn` with no tool call.

Options:
- **(a) Treat `end_turn` without `respond_to_user` as a valid response.** Stream the text directly.
- **(b) Force `respond_to_user` via system prompt rules.** Risk: Claude ignores the rule occasionally.
- **(c) Use `tool_choice: "any"` to force tool use.** Risk: Claude can't emit natural text before the tool call, and it's incompatible with extended thinking.

**Recommendation:** Option (a) — handle both exit conditions. Text-only responses are valid for simple acknowledgments ("Got it, I'll check that for you").
